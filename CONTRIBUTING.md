# Contributing

Thank you for your interest in contributing to this project! This document provides guidelines and instructions for contributing.

## Development Workflow

### Branch Strategy

1. **Main branch** (`main`) — Always in a releasable state
2. **Feature branches** — Created from `main` for new features, fixes, or workflow changes

### Creating a Branch

```bash
git checkout main
git pull origin main
git checkout -b feature/NNN-your-feature-slug
```

Branch naming uses a **global numeric prefix** (`NNN`) shared across features, fixes, and workflow changes. Use `scripts/next-issue-number.sh` to reserve the next available number.

| Prefix | Example | Use case |
|--------|---------|----------|
| `feature/` | `feature/001-dashboard` | New features |
| `fix/` | `fix/002-scope3-total` | Bug fixes |
| `workflow/` | `workflow/003-agent-update` | Agent/workflow changes |
| `docs/` | `docs/update-readme` | Documentation only (no numeric prefix needed) |
| `chore/` | `chore/deps-update` | Maintenance tasks (no numeric prefix needed) |

## AI Agent Workflow

This project uses **GitHub Copilot agents** to automate the development lifecycle. See [`docs/agents.md`](docs/agents.md) for the complete workflow documentation.

### How It Works

A human **Maintainer** coordinates specialized AI agents, each responsible for a stage:

1. **Requirements Engineer** → gathers and documents feature specifications
2. **Architect** → designs technical solution, produces ADRs
3. **Quality Engineer** → defines test plan and acceptance criteria
4. **Task Planner** → breaks architecture into actionable user stories
5. **Developer** → implements code and tests
6. **Technical Writer** → updates documentation
7. **Code Reviewer** → reviews code quality and standards
8. **UAT Tester** → validates user-facing features via real PRs
9. **Release Manager** → creates PR, manages release, publishes
10. **Retrospective** → post-release workflow improvements

Supporting agents: **Issue Analyst** (bug investigation), **Workflow Engineer** (meta-workflow improvements), **Workflow Orchestrator** (end-to-end automation).

### Automated Orchestration

For well-defined features or bugs, assign a GitHub issue to `@copilot` to trigger the **Workflow Orchestrator**, which delegates to all required agents in sequence.

### Agent Definitions and Skills

- **Agent definitions**: `.github/agents/*.agent.md` — role instructions for each agent
- **Agent skills**: `.github/skills/*/SKILL.md` — reusable procedures (PR creation, testing, UAT, etc.)
- **Workflow docs**: `docs/agents.md` — complete workflow, roles, and handoff relationships

## Testing Requirements

All changes must include appropriate tests.

### Running Tests

```bash
# Run all tests
npm test

# Run tests in watch mode
npm run test:watch

# Run a specific test file
npx vitest run path/to/test.test.ts

# Type check
npm run type-check

# Lint
npm run lint

# Build (also validates types)
npm run build
```

### Test Types

- **Unit tests** — Test individual functions, API route handlers, and business logic
- **Integration tests** — Test end-to-end workflows (API → DB → response)
- **Smoke tests** — Ensure `next build` succeeds and key pages render

Tests use **Vitest** and live alongside the code they test (e.g., `lib/calculations.test.ts`).

## Commit Messages

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```text
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

### Types

| Type | Description |
|------|-------------|
| `feat` | New feature (triggers version bump) |
| `fix` | Bug fix (triggers version bump) |
| `docs` | Documentation only |
| `style` | Code style (formatting, no logic change) |
| `refactor` | Code refactoring |
| `perf` | Performance improvement |
| `test` | Adding/fixing tests |
| `build` | Build system or dependencies |
| `ci` | CI configuration |
| `chore` | Maintenance tasks |
| `revert` | Revert a previous commit |

### Examples

```text
feat: add supplier public form page
fix: correct Scope 3 proxy calculation
docs: update API documentation
test: add dashboard KPI tests
chore(deps): update prisma to 6.x
```

### Commit Guardrails

PRs that only change internal tooling (`.github/`, `scripts/`, `docs/`) must **not** use `feat:` or `fix:`. Use `chore:`, `ci:`, or `docs:` instead. This prevents unintended version bumps.

**CHANGELOG.md** is never edited manually — entries are generated by `commit-and-tag-version` in CI.

## Code Style

- **TypeScript** strict mode everywhere
- **ESLint** + **Prettier** for formatting (enforced by pre-commit hooks)
- **Prisma** for all database access (no raw SQL)
- Prefer `const` over `let`; never use `var`
- Use named exports (except Next.js page/layout conventions)
- Prefer `async/await` over raw Promises
- Use functional React components with hooks
- Use Next.js App Router conventions (`layout.tsx`, `page.tsx`, `route.ts`)
- Use server components by default; add `"use client"` only when needed
- Files should not exceed 200–300 lines; refactor at that point

### Pre-commit Hooks

Husky runs `lint-staged` on commit to enforce formatting via ESLint and Prettier.

## Pull Request Process

1. Create a feature branch from latest `main`
2. Make changes with conventional commits
3. Ensure all tests pass: `npm test`
4. Ensure build succeeds: `npm run build`
5. Create PR using `scripts/pr-github.sh create`
6. Wait for PR Validation workflow to pass
7. Merge using rebase-and-merge strategy (linear history)

### PR Description Template

```markdown
## Problem
<why is this change needed?>

## Change
<what changed?>

## Verification
<how was it validated?>
```
